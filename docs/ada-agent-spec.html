Ada is your local chief-of-staff. She lives in the Windows system tray, opens on a hotkey, listens and speaks, and can actually do things on your machine — read and write files, run commands, search the web, remember what matters — but she asks before anything consequential. 2.Where I stand — point of view, hard truths, the moves

2.1 What is genuinely right about this

  • Local-first is the whole product, not a setting. A personal agent earns trust by keeping your life on your machine. The hybrid choice — small model local, frontier model on escalation — means the default path for "tidy these files", "what's on today", "remember this" never makes a network call. Privacy is the architecture, not a checkbox.
  • The harness is the moat. Anyone can wrap a chat model. The hard, valuable part is the environment: approvals, scoping, memory, compaction, audit. The Agent Framework gives us those as first-class pieces. Building Ada well means building the harness well — and that harness is the reusable asset you asked for.
  • Voice being yours (Voxa) is a real edge. Most agents bolt on a cloud STT/TTS. Voxa already gives the .NET pipeline, local STT/TTS options, Microsoft Agent Framework adapter, observability, and Voxa Studio; Simi mobile already speaks the consuming WSS contract. Ada should use that stack directly and keep voice private by default.

2.2 Three hard truths the easy version underweights

Hard truth #1 — an agent that can run shell is a loaded tool
"Full shell + files with approvals" is the right power level, but the approval UX is the safety system. If approvals are noisy, you will click "approve" on autopilot and the gate is theatre. The spec therefore treats the approval policy — risk tiers, allow-lists, batching, "remember this for the session" — as a designed product surface (§8), not a yes/no dialog. Get this wrong and Ada is either annoying or dangerous.
Hard truth #2 — local models are weaker, and tool-calling is where they crack
A 7–14B local model will reason adequately but call tools unreliably and hallucinate arguments. The router (§6) cannot be "use local until it's slow"; it must escalate on task shape — multi-step tool plans, code, long context, anything destructive — to the frontier model, while keeping chit-chat and simple recall local. Honest expectation: in v1, anything involving more than one tool call probably runs in the cloud. That is fine, if it is deliberate and visible.
Hard truth #3 — "shippable soon" raises the floor on everything boring
You chose to build it so others can run it. That means no hard-coded paths, secrets in the OS vault not a .env, a real first-run that installs local models, an MSIX with autostart, and settings a non-engineer can use. None of that is the fun part; all of it is the difference between a demo and a thing. The build plan front-loads the skeleton so the boring floor is poured early.

2.3 The moves that make Ada more than a wrapper

  1. Make approvals legible, not modal. Every gated action is a card with the exact command, the working directory, and a one-line plain-English "why". Approvals can be remembered per-session and per-tool. The audit log is a first-class screen, not a file nobody opens. Trust compounds when you can always see what she did and what she asked.
  2. Memory you can read. Ada's long-term memory is human-readable files with frontmatter and a single index — the same shape this very assistant uses. Nothing is remembered that you cannot open, edit, or delete. Memory is a folder, not a black-box vector blob. Cross-session recall is SQLite FTS5 full-text search over those files (the Hermes Agent approach), with a semantic/vector index left as a maybe-later layer over them, never instead of them.
  3. Draw the AONIK seam now. A "skill" is a self-contained bundle of instructions + tools + (optional) an MCP mount. Today the app reaches AONIK through REST; the future finance skill is just a skill whose tools come from an AONIK MCP server when that surface exists. If the skill contract is right in v1, that integration is configuration, not surgery. We prove the contract by shipping two real skills (research, desktop) against it.

4.The architecture — four planes, the process model, two sequences

4.1 The four planes

Ada is one process, organised as four planes. Each plane has one job and a clean contract to the next. Read top to bottom: input flows down, action flows back up.

UI plane — Ada.App The Windows tray host. A NotifyIcon + a global hotkey + a WebView2 window that renders chat, voice state, and approval cards. Native shell, web-tech surface. Owns notifications, autostart, first-run, settings.
▲ ▼
Voice plane — Ada.Voice In-process ASP.NET Core hosting Voxa (AddVoxa + MapVoxaVoice("/voice")). Voxa runs the loopback desktop endpoint: mic → VAD → STT → the Ada agent → sentence aggregation → TTS → speaker, using local WhisperCpp/Piper/Kokoro/Silero providers by default. Simi mobile proves the consuming WSS contract.
▲ ▼
Agent & harness plane — Ada.Core The heart. A ChatClientAgent wrapped in the harness: the hybrid model router, the approval gateway, compaction, memory, the tool / skill / MCP registry, the audit log. Everything below the UI and voice is here.
▲ ▼
Capability plane — Ada.Tools / Ada.Mcp / providers The actual powers: shell, filesystem, web search/fetch, clipboard, screenshot, scheduler; MCP clients (filesystem, git, and later AONIK); and the model providers (configured local IChatClient by default, Anthropic/OpenAI/Azure on escalation).
Why one process
Voxa hosts in-process and resolves the same singleton Ada agent by DI. Keeping the tray host, the voice WebSocket, and the agent in one .NET 10 process means one lifetime, one DI container, one audit log, one place secrets live, and a loopback-only socket that never listens on a public interface. The WebView2 window is the only "client", and it talks to 127.0.0.1.

4.2 The process & threading model

  • Generic Host (Microsoft.Extensions.Hosting) owns the lifetime: DI, configuration, logging, the voice endpoint, and a tray IHostedService. The app is a Windows desktop app (net10.0-windows) that also runs an in-process Kestrel bound to loopback.
  • The UI thread runs the tray and WebView2. It never blocks on the agent; all agent calls are async and stream back over the voice/IPC channel.
  • One session per conversation. An AgentSession is the runtime handle used for multi-turn RunAsync calls. Persistence is Ada-owned through a local conversation store / ChatHistoryProvider adapter; do not assume preview internal thread serialization.
  • Approvals suspend the run, not the thread. A gated tool call returns control to the host with a pending request; the UI resolves it; the host resumes the same session. The model is never left "hanging" on a socket.

5.2 Composing Ada — the one factory that builds the agent

This is the spine of Ada.Core. Read it as the contract: a hybrid model, a compaction pipeline, a memory provider, an audit middleware, the persona, and the resolved tool set.

Ada.Core/AdaAgentFactory.csC#
// #pragma warning disable MAAI001  — compaction framework is experimental
using Microsoft.Agents.AI;
using Microsoft.Agents.AI.Compaction;
using Microsoft.Extensions.AI;

public static AIAgent Build(AdaOptions o, IServiceProvider sp)
{
    // 1 — the model plane: local-first, cloud on escalation (see §6)
    IChatClient model = new HybridChatClient(
        local:  o.LocalModel,        // configured local IChatClient (Foundry Local if verified)
        cloud:  o.CloudModel,        // Anthropic / OpenAI IChatClient
        policy: sp.GetRequiredService<IRoutingPolicy>());

    // 2 — a small, cheap client used only to summarise during compaction
    IChatClient summarizer = o.SmallLocalModel;

    // 3 — the compaction pipeline: gentle → aggressive (see §8.4)
    var compaction = new PipelineCompactionStrategy(
        new ToolResultCompactionStrategy(CompactionTriggers.TokensExceed(8_000)),
        new SummarizationCompactionStrategy(summarizer, CompactionTriggers.TokensExceed(40_000)),
        new SlidingWindowCompactionStrategy(CompactionTriggers.TurnsExceed(40)),
        new TruncationCompactionStrategy(CompactionTriggers.TokensExceed(120_000)));

    // 4 — build the agent: harness providers on the BUILDER so they run inside
    //     the tool-calling loop; persona + resolved tools via ChatClientAgentOptions
    AIAgent ada = model
        .AsBuilder()
        .UseAIContextProviders(
            new CompactionProvider(compaction),
            new MemoryContextProvider(sp.GetRequiredService<IMemoryStore>()))
        .Use(sp.GetRequiredService<AuditMiddleware>().Invoke)   // log every tool call + decision
        .BuildAIAgent(new ChatClientAgentOptions
        {
            Name = "Ada",
            ChatOptions = new ChatOptions
            {
                Instructions = AdaPersona.SystemPrompt,           // §3.2 in one prompt
                Tools = sp.GetRequiredService<IToolRegistry>().Resolve(), // §7
            },
            ChatHistoryProvider = sp.GetRequiredService<AdaChatHistoryProvider>(), // Ada-owned persistence
        });

    return ada;
}

5.4 Insulating against preview drift

Ada.Core exposes one interface to the rest of the app — IAdaConversation (start a session, send text, stream responses, surface and resolve approvals). Every Agent Framework type stays behind it. When a preview reshapes AgentResponse, approval content, compaction triggers, or history providers, exactly one adapter changes and the UI, voice, and tools are untouched. This is the cost of building on a moving framework, paid once.

6.The model plane — providers, authentication, and the hybrid router

6.1 Providers, not hardcoded clients

Ada ships an opinionated shape — local model by default, cloud model only on escalation — but doesn't bake in a vendor. The Microsoft Agent Framework C# contract is provider-agnostic: any backend that yields a Microsoft.Extensions.AI.IChatClient can back ChatClientAgent. Which models you can reach, and how you prove you're allowed to, sits behind a registry: a catalog of providers over a provider-agnostic client, where swapping or adding a backend is config, not code. Foundry Local remains the preferred local target if its C# bridge and model catalog are verified; otherwise the local default is an OpenAI-compatible runtime on loopback (Ollama / LM Studio / similar).

6.4 Provider-specific auth research — the Claude case

The useful future case: let someone use Claude (or Copilot, or Vertex) without hand-managing an API key, by leaning on a subscription flow or CLI they've already authenticated. That is desirable, but it is not grounded by the Microsoft Agent Framework C# docs and should not be a v1 dependency. Ada treats these as research modes behind feature flags:

  1. Subscription OAuth, run by Ada (auth: oauth-research) — only if the provider publishes a supported app/device flow, refresh semantics, and terms that allow this use. Tokens still land in the OS vault.
  2. Reuse an installed CLI login (auth: cli-research) — only if the CLI exposes a stable command/API for obtaining delegated credentials. Ada must never scrape private token files or depend on undocumented credential formats.
  3. CLI-as-backend — for a provider that only exposes itself through its CLI, Ada can run that CLI as a subprocess behind an IChatClient adapter. This is a compatibility escape hatch with latency and security tradeoffs, not the default path.

Azure is the exception because it already has a supported SDK credential chain. For packaged builds, prefer an explicit Azure credential configuration over broad production use of DefaultAzureCredential.

The rule of thumb
Ship API key, none/local, and Azure credential. Research subscription OAuth and CLI reuse only after confirming the provider's public contract. In every mode, no secret goes to plaintext config, memory files, or the repo.

8.The harness — approvals, plan review, scoping, sandboxing, compaction, observability

8.1 The approval flow, exactly

An approval-required tool does not run when the model asks for it. The run returns with a FunctionApprovalRequestContent; the host shows a card; your decision is sent back on the same session as a FunctionApprovalResponseContent; the run resumes. The loop repeats until no requests remain.

Ada.Core/Harness/ApprovalLoop.csC#
AgentSession session = await ada.CreateSessionAsync();
AgentResponse response = await ada.RunAsync(userText, session);

while (true)
{
    var requests = response.Messages
        .SelectMany(m => m.Contents)
        .OfType<FunctionApprovalRequestContent>()
        .ToList();

    if (requests.Count == 0) break;          // final answer reached

    var replies = new List<AIContent>();
    foreach (var req in requests)
    {
        // Policy may auto-approve (allow-listed / remembered this session)…
        ApprovalDecision d = policy.Evaluate(req.FunctionCall);
        bool approved = d.Auto ?? await ui.AskAsync(req.FunctionCall, d);  // …else ask you
        audit.Record(req.FunctionCall, approved, d.Reason);
        replies.Add(req.CreateResponse(approved));          // true ⇒ run, false ⇒ skip
    }

    response = await ada.RunAsync(new ChatMessage(ChatRole.User, replies), session);
}

8.2 The approval policy — four risk tiers

Not every action deserves the same friction. The policy classifies each tool call and decides whether to auto-allow, ask, or always-ask.

Tier 0 — Read
Read file, list dir, recall, search, screenshot-off. Never gated. No mutation, no egress.
Tier 1 — Local write
Write/move files in an allowed root, clipboard write, schedule reminder. Gated; "remember for session" allowed.
Tier 2 — Powerful
Shell, delete, install, anything outside allowed roots, screenshot. Always gated; session-remember scoped tightly.
Tier 3 — Egress
Web fetch, cloud escalation, remote MCP, any "send". Flagged + gated when it also writes externally.
  • Allow-lists. Roots Ada may write to without asking (e.g. ~/Projects/Ada, ~/Downloads) are configured; everything else is gated. Allowed shell commands (e.g. git status, ls) can be pre-approved as read-only.
  • Session-remember. "Approve, and don't ask again this session" is scoped to the exact tool + argument shape (e.g. "moves into /notes"), and evaporates when the session ends. Never global, never persisted silently.
  • Deny is a first-class answer. Denying returns a rejection to the model, which adapts ("ok, I'll leave those in place") rather than failing.

8.3 Scoping & the blast radius

Approvals are the human gate; scoping is the machine gate, and it holds even if a prompt is adversarial. Ada runs with: a configured set of writable roots; a denylist of never-touch paths (system folders, the vault, her own memory unless via the memory tools, the credential store); a per-call size/time cap on shell; and no elevation — she runs as you, never as admin. The filesystem MCP server is itself launched scoped to a single root, so even MCP-issued file ops cannot wander.

8.4 Compaction — staying inside the window over long sessions

Long voice or work sessions overflow context. The compaction pipeline (registered on the builder in §5.2, so it runs inside the tool loop) trims from gentle to aggressive, preserving system + recent turns. Verbatim shape:

Ada.Core — compaction pipeline (experimental: MAAI001)C#
new PipelineCompactionStrategy(
    new ToolResultCompactionStrategy(CompactionTriggers.TokensExceed(8_000)),   // collapse old tool spew
    new SummarizationCompactionStrategy(summarizer, CompactionTriggers.TokensExceed(40_000)), // summarise mid-history
    new SlidingWindowCompactionStrategy(CompactionTriggers.TurnsExceed(40)),     // keep recent turns

Numbers are tuned per model: the cloud frontier model gets a large budget; the local model's thresholds are scaled down to its smaller window. Compaction summaries use the cheap local summarizer, never the cloud.

8.5 Observability & the audit log

  • Audit log — every tool call, its arguments, the approval decision and reason, the route taken (local/cloud), and any egress, appended to a local, structured log and shown as a settings screen. This is the "what did Ada do" ledger.
  • OpenTelemetry — Ada emits its own spans/metrics for model routing, tool calls, approvals, and compaction; Voxa contributes VoxaActivities.Source and the voxa.turn.ttfb voice metric. Ada wires a local exporter; nothing is sent anywhere by default. A trace viewer in settings helps you (and us) debug latency and tool loops.
  • No telemetry home. v1 ships with zero outbound product telemetry. If that ever changes it is opt-in and itself an egress channel (§14).

8.6 Plan review — approve the strategy, not just the step

The approval loop in §8.1 gates each mutating call as it happens — exactly right for one action, but for a genuinely multi-step task it becomes a stutter of cards, the failure mode behind approval fatigue (§19.2). Borrowing the human-in-the-loop plan gate from ByteDance's DeerFlow: when the router escalates a multi-step or mutating task, Ada first proposes a plan — the ordered steps she intends — and you approve, edit, or reject the whole plan before any execution. The per-tool gates (§8.1) then fire only for the genuinely destructive steps inside it.

Plan review · 3 steps · 1 will need approval
1. Find this week's screenshots in Downloads · read
2. Create ~/Projects/Ada/notes/2026-06 · write
3. Move 7 files into it · write · gated
Approve planEdit…Reject
  • Edit is natural language. "Leave the PDFs where they are" sends the plan back for revision (DeerFlow's [EDIT_PLAN]) instead of forcing a step-by-step denial.
  • Single-step tasks skip it. A one-action request keeps the single approval card from §8.1 — no plan ceremony for "move this one file" (DeerFlow's has_enough_context short-circuit).
  • The plan is legible and logged. It lands in history and the audit log (§8.5) like any other action, so "what did Ada intend" is always recoverable.

8.7 Loop & step budgets — the runaway guard

An action agent can loop: re-calling tools, re-planning, or spinning — burning tokens and money, and (for an unattended job, §12.2) doing it while no one watches. The harness caps every run with three budgets, and on a breach it degrades gracefully rather than crashing or looping: it stops, summarises from what it has so far, and surfaces that — the recursion-fallback pattern from DeerFlow.

BudgetCapsOn breach

Tokens / wall-clockPer-turn ceiling; tighter for unattended jobsDeliver partial; queue it if unattended

That is the agent-loop guard. Its counterpart at the metal is the in-process sandbox (§8.8), where Wasmtime fuel bounds CPU deterministically — verified in the spike, where an infinite loop trapped as OutOfFuel instead of hanging. Loop safety therefore exists at two levels: the agent loop here, and the code sandbox below.

8.8 The sandbox & autonomy ladder

§8.1–§8.3 make the host safe by gating every mutation. But real autonomy — letting Ada run agent-written code, crunch data, drive a browser, or process untrusted content without a gate on every step — needs a stronger boundary than per-action approval. The move is to give her a disposable zone to work in: inside it she acts freely; the only gate is the zone boundary — writing a result back to your real machine, or egress. Done right, autonomy widens while the blast radius shrinks to a throwaway sandbox.

Zone 0 — Host
Your real machine. Stays gated/scoped exactly as §8.1–§8.3. Where "tidy my real Downloads" runs. Everyone.
Zone 1 — In-process Wasm · default
Untrusted code in-process via wasmtime-dotnet with capability isolation + fuel + memory limits. No dependency — ships in the MSIX, runs on Windows Home. TS/JS + scripting-Python.
Zone 2 — Local container · opt-in
The AIO Sandbox over Docker/WSL2: real browser, shell, data-science Python. Detected & recommended by the wizard (§11.3).
Zone 3 — Remote (Azure) · future
The same AIO Sandbox on Azure Container Apps. Full power, zero local footprint. Post-v1.
The rule that buys safe autonomy
Inside a zone, mutating work needs no per-step approval — nothing there can hurt you. The gate moves to the boundary: materialising a sandbox result onto the real disk, or any egress (§14.1). That is what lets an unattended job (§12.2) do real multi-step work overnight and queue only the boundary crossing for review — and it turns prompt injection (§14.4) from a catastrophe into a discarded container.

All four zones sit behind one seam, so the agent asks to "run code" and never names a backend — the same insulation discipline as §5.4. The default backend is the verified in-process Wasm one; the others swap in by config.

Two product rules carry straight over from Simi's Keeper and are baked into the skill's instruction fragment: amounts keep their original currency, never converted; and Ada describes, never prescribes — she'll tell you what you committed and what you've paid, never what to do with your money. Memory links ([[simi-records]]) are already laid so Ada knows the seam exists before the skill does. Nothing about this is in the v1 build — only the contract it will use.

9.5 Recall across sessions — FTS5, not vectors

For "what did we decide about X last week?", Ada searches her own history. Following Hermes Agent's lead, this is SQLite FTS5 full-text search over sessions and memory, with the model summarising the hits — not a vector database. It's local, fast, zero extra infrastructure, and inspectable. A semantic/vector index stays a maybe-later optimisation layered over the same files, never a replacement for them.

10.2 Behaviours that make voice usable

  • Interruption / barge-in, explicitly tested. Voxa supports priority interruption frames and server-side barge-in purge, so queued bot audio can be dropped when the user speaks. 12.Automations & schedules — cron, unattended runs, delivery

The morning briefing in §3.5 is, quietly, a cron job — and §7's Scheduler tools so far cover only passive reminders (fire a toast at a time). This section specifies the real thing: autonomous scheduled runs, where Ada wakes on a cadence to actually do work. It's what makes her proactive rather than reactive — and it's the one capability that collides head-on with "ask before you act," because a scheduled run may fire when you're not there to approve. The design borrows from Hermes Agent's built-in cron: natural-language schedules, a delivery target per job, and unattended runs paired with command allow-lists.

12.2 What an unattended run may do (the crux)

A scheduled run obeys a stricter version of the §8 approval policy, because you may be asleep:

  • Read-only by default. Unattended runs may gather, read, summarise, draft, and remember — Tier-0 / Tier-1-read work. They may not execute any gated (mutating / egress-write) action on their own.
  • Draft-and-queue. If a job wants a gated action it stops and queues it as a pending approval with the full card; you clear the queue when you're back ("Ada prepared 3 actions overnight — review"). The work is done up to the gate, not past it.
  • Standing grants (opt-in, narrow, revocable). For trusted recurring work you can pre-authorise a specific action shape once — "file screenshots into /notes every Friday" — scoped to that tool + argument pattern, listed in settings, killable, fully audited. This is Hermes's command allow-list, kept deliberately tight.
  • Contained autonomy (with a sandbox). If a sandbox zone is enabled (§8.8), an unattended job may run mutating work freely inside the zone — analyse, build, transform — and queue only the boundary crossing (a write to your real disk, or egress) for review. The blast radius is a disposable sandbox, so overnight autonomy widens without widening risk.
The default
Unattended = read-only + draft-and-queue. Autonomy over mutating actions is something you grant per-job, narrowly, on purpose — never the default. A single kill switch in the tray pauses every automation at once.

12.3 Natural-language schedules, delivery where it makes sense

You create a job by describing it — "every weekday at 8, brief me on today" — and Ada derives the schedule (a cron expression you can see and edit). Every job has a delivery target for its result:

Delivery targetUse
Toast notificationA glanceable nudge ("3 PRs need review")
The chat windowA full result waiting when you next open Ada
A written note (file)A durable artefact — the morning brief saved to /notes
Later a channel (email / WhatsApp)Hermes delivers to messaging platforms; for Ada this bridges to Simi's "WhatsApp is a channel" idea — out of scope for v1.

12.4 Triggers — time now, events later

v1 triggers are time-based (cron) plus at-login. The trigger is an abstraction, though, so later additions — an inbound message, a file appearing, the machine going idle — slot in without reworking the engine. (Hermes's persistent "gateway" shows the value of event triggers; Ada defers them, but designs for them.)

12.5 The runtime — firing when Ada isn't open

A closed tray app, or a sleeping laptop, can't fire an in-process timer. So the runtime is a hybrid:

  • In-process scheduler (a hosted, Quartz-style service) handles jobs while Ada is running.
  • Windows Task Scheduler registration wakes or launches a headless Ada for time triggers when she isn't open — with "run if missed" catch-up and survive-reboot semantics.
  • An unattended run executes in a background session, distinct from interactive chat and tagged as autonomous in history and the audit log, so "what Ada did at 03:00" is always legible.

12.6 Cost & egress when you're not watching

An unattended job that escalates to the cloud spends money and egresses data with no one watching. So scheduled jobs default to the local model (§6); cloud escalation for a job is opt-in per job with a token/cost budget and, like every escalation, a logged egress (§14). Web and remote-MCP egress inside a job follow the same per-job opt-in.

14.Security, privacy & trust — the egress contract

14.1 The egress contract (the heart of "private by default")

Ada names every way data can leave the machine, and treats each as a channel that is listed, toggleable, and logged. If a channel is off, that capability simply doesn't exist that session.

14.2 Secrets

API keys and MCP auth tokens live in the Windows Credential Manager / DPAPI, never in config.json, never in memory files, never in the repo (the full resolution order is in §6.2). Azure credentials use the configured Azure credential chain rather than Ada storing a secret. Research OAuth/CLI modes, if later enabled, must follow the same rule: supported provider flow only, no private token scraping, no plaintext storage. Everything is read at call time and never written to logs — the audit log redacts known secret-shaped values — and the vault is on the never-touch denylist (§8.3).

14.3 The blast-radius guarantees

  • Ada runs as you, never elevated; she cannot install system-wide or touch another user's data.
  • Writes are confined to allowed roots; the denylist protects system folders, the vault, and her own memory store (editable only through memory tools).
  • Shell calls are capped (time + output size), gated, and run in the user context with no implicit network trust.
  • The voice WebSocket binds to 127.0.0.1 on an OS-assigned port; it never listens publicly.
  • The model cannot bypass a gate. Approval enforcement is in the host loop (§8.1), not a polite request in the prompt.

14.4 The threat model, briefly

The realistic risk for an action-taking agent is prompt injection via content she reads (a web page or file that says "delete everything"). Defences: mutation is always gated and you see the literal command; scoping bounds the damage even if you mis-approve; egress is named so exfiltration attempts are visible; and the router sends anything destructive to the stronger model, which resists injection better. Ada is honest that this risk exists and designs so a single bad click is survivable, not catastrophic.
