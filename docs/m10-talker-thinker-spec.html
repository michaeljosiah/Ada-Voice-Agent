Ada · Specification M10 · July 2026

The Talker/Thinker Split.

Ada's voice today goes silent while her tools work — a sandbox run, a web search, a mail sweep is fifteen seconds of dead air. Voxa's VDX-008 background agent delegation fixes exactly this: a fast talker keeps the conversation while a heavyweight thinker runs the tools off the latency path, and the result re-enters as a new, relevance-gated turn. This spec is Ada's adoption plan — including the one wrinkle that shapes everything: Ada hand-builds her Voxa chain, so the advertised one-line opt-in doesn't apply until she either wires the Core pieces herself (Path A) or takes the long-deferred VDX-007 migration first (Path B, recommended).

Implementation refinements (v1.1 — what shipped differs from the draft in four places). (1) Delegation is heuristic, not a model tool call: Ada's tool mode is skill-forced (load_skill), so a bare delegate_task function would rarely be called by a local model. Instead the engine's existing ShouldUseTools() heuristic — which identifies exactly the turns that stall voice — became the trigger: a voice turn (AdaRequest.AllowDelegation) that would enter tool mode yields a Delegate chunk plus a canned acknowledgment, and the thinker runs the goal with the full tool harness in the background. Text surfaces never set the flag (M10-T7 by construction). (2) Result deliveries are ChatOnly: a new AdaRequest flag pins background-result turns to the fast chat agent so their content can never keyword-match back into tool mode. (3) Settings → Voice under UseDefaults(): a live configuration source keeps STT model/language and voice switches applying on the next session; switching the TTS provider (Piper ↔ Kokoro) now needs an app restart (Voxa:Tts binds once into VoxaOptions). (4) The thinker is a keyed second engine (AdaCoreServiceCollectionExtensions.ThinkerEngineKey) on the escalation client with its own in-process history, and gated tools fail fast inside ToolContext.EnterBackgroundScope() (AsyncLocal) rather than a policy object.

1. What is this, in plain English?

When you ask Ada by voice to "check my mail for anything from the bank," the M6 voice loop pipes the transcript into IAdaEngine.RespondAsync and waits. The engine picks its model, runs the mail tool, composes an answer — and for that whole stretch the room is silent. In text chat that's a spinner; in voice it reads as a hang.

VDX-008 (merged to Voxa main 2026-07-10, spec v1.2) splits the agent in two. The talker answers instantly — "on it, give me a moment" — and hands the slow work to a thinker via an explicit delegate_task tool call. The thinker runs off the voice-latency path inside a new BackgroundAgentProcessor; when it finishes, its result re-enters the agent loop as a new turn that the talker gates for relevance (it may stay silent if the conversation moved on). Results are never injected while the user is speaking, barge-in never cancels delegated work, and a failed or timed-out task comes back as an apologizable error rather than silence.

For Ada the split is unusually natural: her hybrid router already distinguishes a fast local path from a heavier cloud path. The talker is the fast path with no tools; the thinker is the cloud path with the full tool harness.

2. The wrinkle: Ada is not on UseDefaults()

Voxa's advertised adoption is one line — services.AddVoxaBackgroundAgent(…) — plus a Voxa:BackgroundAgent:* config block. That line works by a keyed-DI seam inside DefaultVoicePipelineComposer. But AdaVoice.MapAdaVoice never calls the composer: it hand-builds the chain processor-by-processor (InputRateTagProcessor, Silero, WhisperCpp, filters, a hand-constructed AgentLoopProcessor, TurnEndSentenceAggregator, TTS) — the M6.1 fidelity workarounds, most of which Voxa has since absorbed upstream. The composer seam therefore does not fire for Ada, and neither does the MAF delegate_task auto-injection (Ada's tools live inside IAdaEngine, not in a MAF agent Voxa can see).

Consequence. Ada integrates at the Core level: construct BackgroundAgentProcessor herself, arm AgentLoopProcessor herself, and emit/consume the delegation frames from her own driver. That is Path A. Or she first takes the VDX-007 migration that was speced for her (register IAgentTurnDriver in DI, keep UseDefaults()), after which VDX-008 really is one registration plus config. That is Path B.

3. Work common to both paths (the real substance)

3.1 A background driver — the thinker

A ~20-line AdaBackgroundTurnDriver : IAgentTurnDriver mirroring AdaEngineTurnDriver, with three deliberate differences:

  • Forced heavyweight route. It should pin the engine to the cloud path with the full tool harness (the router's escalation heuristics don't apply — delegation is the escalation decision).
  • Its own thread — or none. Research traces must not pollute the voice conversation's thread. The result text is what matters; if it's worth remembering, the talker's spoken reply lands in the thread anyway.
  • No sanitizer. Its text is read by the talker, not spoken; skip VoiceTextSanitizer. Yielding plain LlmTextChunkFrames is enough — Voxa's whitelist accumulates them into ResultText and guarantees they never reach TTS directly. Optionally yield StatusFrames ("Searching your mail…") for progress UI.

3.2 The delegation signal — keeping Ada.Core Voxa-free

The talker must emit BackgroundTaskRequestFrame, but Ada's delegate_task tool would live in the engine's catalog inside Ada.Core — which correctly has no Voxa reference and must stay that way. Two clean options:

OptionMechanicsVerdict
New chunk kind The engine's delegate_task tool returns the ack string to the model and emits an AdaResponseChunkKind.Delegate chunk carrying goal + contextSummary. AdaEngineTurnDriver — which already switches on Kind — translates it into yield return new BackgroundTaskRequestFrame(taskId, goal, context, ctx.TurnId). recommended — fits Ada's chunk-stream architecture; the CLI and text surfaces simply ignore the new kind.
Callback on AdaRequest The driver passes a Func<string, string?, ValueTask> wrapping ctx.Emitter.EmitAsync(…); the tool handler invokes it. Workable, but threads a voice-only concern through AdaRequest for every surface.
The ack string carries the contract. Whatever the tool returns to the model must say: "Delegated (task X). The result will arrive shortly as a system note. Acknowledge to the user that you're working on it — do NOT invent the answer." Voxa's experience is that models follow tool results far more reliably than instruction addenda. The talker's system prompt should also gain one line describing when to delegate (anything needing tools or more than a beat of thought).

3.3 Consuming results — the trigger check

AdaEngineTurnDriver.RunTurnAsync gains a branch: on ctx.Trigger == TurnTrigger.BackgroundResult, feed ctx.BackgroundResult.ResultText — framed by the relevance-gate instruction — into the engine instead of ctx.UserText (which is empty on these turns). Ada already references Voxa.Services.MicrosoftAgents, so MicrosoftAgentVoice.CreateBackgroundResultMessage(result).Text supplies the exact upstream wording, including the license to answer with nothing when the conversation has moved on. An empty engine response is a valid outcome — the loop treats it as a silent turn.

Guard the thread factory. ThreadIdForVoiceTurn currently creates a thread titled from ctx.UserText on the first turn. A background-result turn has empty user text — it must not create a thread titled "Voice chat" out of nothing. Check the trigger before creating.

3.4 Approval gates

Voxa throws on ctx.FrontendTools in background turns by design: nobody is looking at an approval dialog for work they delegated and moved past. The thinker's tool policy must therefore be auto-approve only — the pre-approved allowlist, no gated tools. A gated tool reached from a background turn should fail fast into the task's IsError result, which the talker can voice ("that needs your approval — want to do it on screen?").

4. Path A — surgical additions to the hand-built chain

Everything in §3, plus two mechanical insertions in MapAdaVoice:

// after the AgentLoopProcessor stage, before TurnEndSentenceAggregator:
.UseProcessor(() => new BackgroundAgentProcessor(
    new AdaBackgroundTurnDriver(engine, vlog),
    maxConcurrentTasks: 2,
    taskTimeout: TimeSpan.FromSeconds(120)))
// and on the existing AgentLoopProcessor construction:
new AgentLoopProcessor(
    new AdaEngineTurnDriver(engine, ThreadIdForVoiceTurn, vlog),
    ...,
    backgroundResults: new BackgroundResultOptions())   // defaults: hold-while-speaking, 4 pending, 2 s timeout

Config knobs are constructor arguments on this path — the Voxa:BackgroundAgent:* section is a composer feature and does not apply.

5. Path B — take the VDX-007 migration first recommended

The hand-built chain exists for reasons Voxa has since absorbed: BlankTranscriptionFilter is retired by TranscriptionFilter.DropBracketedMarkers (on by default), the 900 ms stop duration is Voxa:Vad:StopDurationMs, and VDX-007's entire purpose was Ada keeping UseDefaults() while owning the agent: 

builder.Services.AddScoped<IAgentTurnDriver>(sp => new AdaEngineTurnDriver(…));   // VDX-007
builder.Services.AddVoxaBackgroundAgent(sp => new AdaBackgroundTurnDriver(…));      // VDX-008
app.MapVoxaVoice("/voice").UseDefaults();
// appsettings: Voxa:Vad:StopDurationMs=900, Voxa:BackgroundAgent:TaskTimeoutSeconds=120, …

§4's insertions vanish; the driver-side work of §3 remains (it is needed on either path). Ada sheds InputRateTagProcessor, BlankTranscriptionFilter, the manual VAD/tuning plumbing, and most of MapAdaVoice — and picks up composer features (diagnostics taps, future stages) for free. The remaining open question for B is whether TurnEndSentenceAggregator's flush-on-turn-end behavior is still needed under the current composer, or whether upstream SentenceAggregator now suffices — verify during the spike; if still needed, it is one IVoiceAgentConfigurator-style follow-up to Voxa rather than a reason to keep the whole manual chain.

7. Acceptance checks

  • M10-T1Ask a question needing a slow tool; the talker acknowledges within the normal turn budget while the tool runs. No dead air > 2 s between transcript-final and first talker audio.
  • M10-T2The thinker's result is spoken as a follow-up turn, and lands in the voice thread's history as Ada's message. Thread shows: user ask → ack → result reply. No empty user messages.
  • M10-T3Interrupt the talker mid-ack; the delegated task still completes and its result still arrives. Barge-in never cancels background work (VDX-008 semantics).
  • M10-T4Result arriving while the user is mid-sentence is held and spoken only after their utterance's turn. Voxa's deferred-to-stop release; no talking over the user.
  • M10-T5Ask, then change the subject before the result lands; the talker may stay silent, and the conversation shows no orphan reply. Relevance gate honored; silent turns leave no thread trace.
  • M10-T6A thinker task that hits a gated tool fails fast into an IsError result the talker voices gracefully. No hung task waiting on an approval dialog nobody sees.
  • M10-T7Text/CLI surfaces are byte-identical — the Delegate chunk kind is ignored outside voice, and nothing changes when no thinker is registered. Ada.Core remains Voxa-free; existing surface tests untouched.

8. Effort

Path A: roughly a day — the chunk-kind plumbing and thread-guard are the substance; the two chain insertions are minutes. Path B: the same §3 work plus the deferred M6.1 cleanup — a larger diff, a smaller Ada. Recommendation stands: B, spiked behind the existing voice feature flag, with the TurnEndSentenceAggregator question answered early.
