Ada · Voice Plane · Fix Specification

Voice Pipeline Fidelity · M6.1

The M6 voice pipeline configures Voxa:Profile=LowLatency and Voxa:Vad:Engine=Silero, then hand-builds a pipeline that honours neither. This spec makes the running pipeline match the configured intent — the real Silero VAD and the LowLatency tuning Ada already asks for — without giving up Ada's per-thread voice memory.

1The defect

Ada's voice plane (AddAdaVoice / MapAdaVoice in src/Ada.Voice/AdaVoice.cs) does everything right at configuration time: it registers the Silero VAD descriptor, sets Voxa:Vad:Engine = "Silero", and seeds Voxa:Profile = "LowLatency" both as a startup default and per-connection. But it then maps the route with MapVoxaVoice("/voice").Use((context, pipeline) => …) and hand-assembles the stages. The hand-assembly bypasses Voxa's DefaultVoicePipelineComposer — the only component that reads the profile and selects the registered VAD — so three things Ada explicitly configures never actually run.

The headline
.UseSilenceGate() builds a bare SilenceGateProcessor() — a fixed-constant RMS energy gate. The Silero engine Ada registers and downloads is never instantiated, and the LowLatency profile is inert. The voice stack the README advertises ("Silero VAD … barge-in") is not the one that runs.

Evidence
UseSilenceGate() → new SilenceGateProcessor() (Voxa SpeechBuilderExtensions.cs); the profile is applied only by DefaultVoicePipelineComposer.Compose(), reached only via .UseDefaults(). new SentenceAggregator() defaults to EagerFirstChunkMinChars = 0 / MaxBufferChars = 500, versus LowLatency's 40 / 350 (VoxaTuningResolver).

2Root cause

The manual chain was chosen for a legitimate reason: Ada needs custom turn persistence. Its BuildMessages / OnTurnCompleted hooks read and write Ada's own IConversationStore keyed by ?thread=, so a spoken follow-up has session context and voice turns show up alongside text. Voxa's .UseDefaults() wires its own in-memory InMemoryChatHistory and exposes no seam to substitute a custom store — so Ada could not use the composer wholesale.

The mistake is not "Ada hand-built a pipeline." It is that hand-building threw away the VAD and the profile tuning along with the agent stage Ada actually needed to customise. Those are separable. The convenience builder methods (UseSilenceGate, UseSentenceAggregator) each call a parameterless constructor; none of them read the profile. The profile lives in VoxaTuningResolver, which is a public, DI-registered service. We can resolve it ourselves and feed the same numbers the composer would.

Load-bearing prerequisite — the sample rate
The browser client captures and downsamples to 16 kHz ("Whisper needs exactly 16 kHz", wwwroot/index.html). But Ada never calls .UseDefaults(), so it never sets the session input rate — and that builder property is internal, so Ada cannot set it. Voxa's WebSocketAudioSource therefore tags every inbound frame with its default 24 kHz. The current energy gate is rate-agnostic, so this never mattered. Silero is not. Its processor silently forwards audio without gating when the frame rate ≠ its configured rate — so dropping Silero in naively yields no VAD at all. The frames must carry their true 16 kHz before the VAD. Fixing the label also removes a latent mislabel that reaches Whisper today.

3Design

Stay on the manual .Use(...) route (keep Ada's custom memory hooks untouched), and make three surgical changes inside the per-connection callback:

  1. Resolve the active profile once. Pull VoxaTuningResolver and IOptions<VoxaOptions> from the request scope and call Resolve(). This returns the same VoxaEffectiveTuning the composer uses — so we honour whatever Voxa:Profile is set (LowLatency today) and any explicit overrides, with no magic constants in Ada.
  2. Fix 1 — the real VAD. Tag inbound frames at their true 16 kHz, then replace .UseSilenceGate() with the registered SileroVadDescriptors.Vad, configured from the resolved tuning. This is exactly the VAD stage the composer builds (preroll, confidence∧RMS, eager-STT, force-split).
  3. Fix 2 — the tuned aggregator. Replace .UseSentenceAggregator() with a SentenceAggregator carrying the resolved EagerFirstChunkMinChars and MaxBufferChars — restoring fast first-audio.

Resolved LowLatency values, for reference: confidence 0.5 · minRMS 0.003 · start 150 ms · stop 400 ms · preroll 300 ms · eager-STT 150 ms · max-utterance 20 s · first-chunk 40 chars · buffer 350 chars.

4The patch

4.1 New file

src/Ada.Voice/InputRateTagProcessor.cs new

A minimal stage that relabels inbound audio to the rate the client actually sends. It reuses the existing PCM buffer (no copy) and is a no-op once a frame already carries the target rate. It mirrors the existing BlankTranscriptionFilter pattern.

InputRateTagProcessor.csC#
using Voxa.Frames;       // Frame, AudioRawFrame
using Voxa.Processors;   // FrameProcessor

namespace Ada.Voice;

/// <summary>
/// Stamps inbound <see cref="AudioRawFrame"/>s with the sample rate the browser actually sends
/// (16 kHz — see wwwroot VOICE_IN_RATE / "Whisper needs exactly 16 kHz"). Ada drives the pipeline
/// through the fluent .Use(...) route, which never sets Voxa's (internal) session input rate, so the
/// WebSocketAudioSource defaults to 24 kHz. The Silero VAD only gates when the frame rate matches its
/// configured rate, so this corrects the label before the VAD — and hands Whisper an honest rate too.
internal sealed class InputRateTagProcessor : FrameProcessor
{
    private readonly int _rate;

    public InputRateTagProcessor(int rate) : base("InputRateTag") => _rate = rate;

    protected override async ValueTask ProcessFrameAsync(Frame frame, CancellationToken ct)
    {
        if (frame is AudioRawFrame a && a.SampleRate != _rate)
            frame = new AudioRawFrame(a.Pcm, _rate, a.Channels); // reuse the same PCM memory
        await PushFrameAsync(frame, ct).ConfigureAwait(false);
    }
}

4.3 The pipeline callback

MapAdaVoice → MapVoxaVoice("/voice").Use(...)

Add a sample-rate constant beside the other voice fields:

AdaVoice.csC#
/// The rate the browser client captures at (wwwroot VOICE_IN_RATE); Whisper's native rate.
private const int ClientInputSampleRate = 16000;

Inside the .Use((context, pipeline) => …) body, after var sp = context.RequestServices;, resolve the profile and build the VAD settings:

AdaVoice.cs — new, before the `pipeline` chainC#
// Honour the configured Voxa:Profile (LowLatency) on the hand-built chain. The composer would do
// this for us under .UseDefaults(); we resolve the same tuning so the manual route stays faithful.
var tuning = sp.GetRequiredService<VoxaTuningResolver>()
               .Resolve(sp.GetRequiredService<IOptions<VoxaOptions>>().Value);

var vadSettings = new VoxaVadSettings(
    SampleRate:          ClientInputSampleRate,
    ConfidenceThreshold: tuning.VadConfidenceThreshold,
    MinRms:              tuning.VadMinRms,
    StartDuration:       tuning.VadStartDuration,
    StopDuration:        tuning.VadStopDuration,
    PrerollDuration:     tuning.VadPrerollDuration)
{
    EagerSttDelay        = tuning.VadEagerSttDelay,
    MaxUtteranceDuration = tuning.VadMaxUtteranceDuration,
};

Then change the head and tail of the pipeline chain:

 pipeline
-    .UseSilenceGate()

The agent stage, the BlankTranscriptionFilter, and the per-thread persistence hooks are untouched. SileroVadDescriptors / WhisperCppDescriptors are already imported.

Adjacent & optional — one more LowLatency knob
LowLatency also caps a runaway response at MaxResponseDuration = 30 s. If desired, set opts.MaxResponseDuration = tuning.MaxResponseDuration; inside the existing UseMicrosoftAgent configure callback. Out of the two-fix core scope, but a one-line completion of profile fidelity.

5Acceptance criteria

Done when
  • Silero actually runs. With ADA_LOG=Debug, a spoken utterance logs SileroVad: gate OPENED … prerolling N windows and a matching gate CLOSED. No received audio at 24000 Hz … forwarding without VAD warning appears.
  • The energy gate is gone. No SilenceGateProcessor is constructed on the voice path (grep / a unit assertion on the composed stage list).
  • The profile is live. The aggregator reports EagerFirstChunkMinChars == 40 and MaxBufferChars == 350 under the default LowLatency profile; switching Voxa:Profile changes them without code edits.
  • First syllables survive. A clipped-onset phrase ("Ada, what time is it?") transcribes its first word — preroll is flushing.
  • No regression in memory. A spoken follow-up still sees prior-turn context and the turn still lands in the ?thread= conversation (the custom hooks are unchanged).
  • Builds clean. dotnet build Ada.slnx with no new warnings.

6Out of scope & follow-ups

  • later Output rate via the session envelope. The client already learns the TTS output rate from Ada's GET /api/voice; this spec does not add a Voxa SessionInfoFrame. Unaffected by the two fixes.
  • later Acoustic echo cancellation. With speakers (not headphones) and no AEC, the bot's own voice can still trip the VAD into a false barge-in. A real VAD narrows this versus the energy gate, but the durable fix is an AEC stage — a separate increment.
  • later Upstream: a composer seam for custom memory. The clean long-term shape is for Voxa's .UseDefaults() to accept a custom chat-history / agent-options hook, so a host with its own conversation store keeps the composer (VAD + profile + diagnostics) instead of re-deriving it. That would retire this spec's manual stages entirely and is worth raising against Voxa. Until then, this spec is the faithful in-Ada fix.
  • non-goal Changing the granular interruption model. Barge-in here is sink-side audio purge (keyed on the speaking edge both gates emit); no in-flight-turn InterruptionFrame is introduced.
