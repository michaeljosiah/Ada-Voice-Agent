Ada · Specification M9 · July 2026

Omnigent as an Optional Substrate.

Ada keeps her body — the voice loop, the memory files, the approval UI, the tray — and gains the option of renting a different brain-executor: an Omnigent server that runs her agent turns on any of ~15 coding harnesses (Claude Code, Codex, Goose, OpenCode, …) with cloud or local sandboxing. Off by default, additive by construction, and gated on measured voice-grade latency.

Document M9 v1.0 Status Draft — spike first Depends on M0–M8 shipped Omnigent pin v0.4.0 (2026-07-03) Source of truth omnigent.ai + repo API.md + openapi.json

1. What is this, in plain English?

Today Ada's brain is a Microsoft Agent Framework ChatClientAgent over an IChatClient: a hybrid router picks a local model (Ollama / ONNX) or escalates to a cloud model, and Ada's own harness runs the loop — tools, skills, memory folding, approval gates, sandboxes. That is the right default and it stays the default.

But some tasks are bigger than a chat loop: "refactor this repo", "triage my downloads folder and write a report", "keep working on this until the tests pass". Purpose-built coding harnesses are far better at long agentic work than a bare chat loop, and Omnigent (Apache-2.0, by the Databricks AI team) is a common REST layer over ~15 of them, plus sandboxed execution (Modal, Daytona, Kubernetes, E2B), session persistence, policies, and human-approval plumbing. M9 lets Ada optionally route a turn — or a whole thread — to an Omnigent server, while everything the user touches stays Ada: her voice, her window, her memory, her approval prompts.

One sentence. M9 adds an OmnigentEngine : IAdaEngine that runs Ada's turns on an Omnigent server over REST + SSE, mapped faithfully onto Ada's existing chunk-stream contract — nothing above the engine seam changes, and Substrate: Local remains the default.

3. Where it plugs into Ada — the seam decision

Ada has three altitudes where a substrate could attach. M9 chooses the top one.

AltitudeSeamVerdict
Engine (chosen) IAdaEngine.RespondAsync(AdaRequest) → IAsyncEnumerable<AdaResponseChunk> — the single seam every host (CLI, WebView2, voice, tests) already talks to. Right. Omnigent owns the loop, the tools, and the history — exactly what an IAdaEngine owns. Every Ada surface inherits the substrate for free, including voice (via the existing AdaEngineTurnDriver).
Model IChatClient inside the hybrid router. Wrong. Omnigent is not a model endpoint; wrapping it as one would run Ada's MAF loop around Omnigent's own loop — double agents, double tools, double history.
Voice stage Voxa's IAgentTurnDriver directly. Too narrow. Only the voice plane would get the substrate; CLI and window turns would not. The engine altitude reaches all of them through the driver Ada already has.

Composition

// Ada.Core — registration (AddAdaCore), config-driven:
// Substrate: "Local" (default) | "Omnigent"
services.AddSingleton<IAdaEngine>(sp => cfg.Substrate switch
{
    AdaSubstrate.Omnigent => new SubstrateRoutingEngine(
        local:    BuildLocalEngine(sp),          // today's AgentEngine, unchanged
        omnigent: new OmnigentEngine(sp.GetRequiredService<OmnigentClient>(), cfg.Omnigent),
        rules:    cfg.Omnigent.Routing),          // per-thread / per-request override
    _ => BuildLocalEngine(sp),
});

SubstrateRoutingEngine is a thin dispatcher: a thread pinned to a substrate stays on it (history lives where the turns ran); an unpinned request follows the routing rules (voice: local by default — §7). The local engine is untouched; with Substrate: Local the composition is byte-identical to M8.

4. The mapping — Omnigent wire ↔ Ada chunk stream

OmnigentEngine translates one AdaRequest into one Omnigent turn and folds the SSE stream back into Ada's chunk contract. The mapping is total: every event family either maps or is deliberately ignored, so a new server-side event can never fault a turn.

Flat payloads. Real response.* frames are flat ({type, delta} / {type, item}); some session.* events are nested envelopes. The Arke spike's first normalizer read a nested shape everywhere and had to be rewritten — Ada's parser is written against API.md's documented shapes and tested against captured frames from the spike (P0), not hand-written fixtures.

Correlation and the queue

Omnigent has no client-message-id echo: correlation between a posted message and stream items is FIFO + session.input.consumed (itself flagged as unstable). OmnigentEngine therefore allows one in-flight turn per session and treats everything between post and response.completed as belonging to that turn — matching Ada's engine contract (one RespondAsync at a time per thread) rather than fighting the wire.

5. Sessions, threads, and reconnect

  • Thread map. One Omnigent session per Ada thread, created lazily on the thread's first Omnigent turn and persisted (threadId → conv_…) in Ada's SQLite store. History for substrate threads lives server-side; Ada's local memory (files + FTS5) still receives the turn summaries it records today, so recall works across substrates.
  • Agent identity. On first use Ada uploads an agent bundle (POST /api/agents) generated from her persona + memory-context preamble. Skills are folded into instructions; Ada's local gated tools are not exported (§6). The bundle is content-hashed; a persona/skill change re-uploads and re-points new sessions.
  • Runner binding. Config selects host_type: "external" + a runner_id on the user's own host (default: the local machine running omni host) or "managed" (server-provisioned sandbox). A session whose runner is offline fails preflight (§8) rather than 503-ing mid-conversation.
  • Reconnect. On SSE drop mid-turn: reopen the stream first, snapshot GET /v1/sessions/{id}, dedupe by item id, and continue folding. If the turn completed while disconnected, synthesize the final from the snapshot's items. If the session is failed, surface a per-turn error chunk.
  • Fork as retry. "Try that differently" maps to POST /v1/sessions/{source}/fork (optionally up_to_response_id + model_override) — a new idle session; re-bind the runner, update the thread map. Deferred to P5; recorded here because the wire supports it cleanly.

6. The security model shift — zones vs. runners

This is the sharpest edge of M9. Ada's M2 safety floor scopes tools to allowed directories, gates writes behind approvals, and sandboxes execution in Zones 0–2 on Ada's machine. An Omnigent turn executes tools on the runner host under Omnigent's policy engine — Ada's zone scoping does not reach it. M9 does not pretend otherwise; it fences the substrate instead.
  • Default fence: a scoped workspace. The default runner binding points at a dedicated workspace directory (workspace on session create), not the user's home. A managed (sandboxed) host is the recommended setting for anything beyond that workspace.
  • Elicitation bridge. Omnigent's human-approval mechanism becomes Ada's: response.elicitation_request surfaces in Ada's existing approval UI (tray toast + window prompt, same as GatedAIFunction approvals), and the user's accept / decline / cancel is posted to …/elicitations/{eid}/resolve. Decline cancels the turn cleanly (v0.4.0 behavior). Because resolve is 202-ack-only, Ada confirms the outcome from the event stream (…resolved / turn continuation) before updating the UI.
  • Policies as belt-and-braces. The agent bundle ships conservative session_policies (spend cap, intent gate on destructive commands) so the fence holds even if the UI is away.
  • Ada's local tools stay home. No fs/shell/web tool of Ada's is exported to the bundle in M9. A later milestone may expose selected Ada capabilities as client-side tools (the waiting + function_call_output round-trip), which would put them back behind Ada's own gates — recorded as future work, not in scope.

7. The voice plane — unchanged, and gated

Voice is where Ada lives, and it is exactly why M9 must not regress latency. Nothing in the voice pipeline changes: Voxa still runs STT/VAD/TTS locally, AdaEngineTurnDriver still yields Answer chunks into LlmTextChunkFrames, and barge-in still cancels the per-turn token — which OmnigentEngine now translates into the wire interrupt (202, queue-bypassing). Omnigent takes no audio (input_audio is "Not Yet"): the substrate always receives final STT text and returns text for local TTS. Omnigent replaces only the brain, never the ears or the voice.

Voice gate. exit criterion Voice turns route to Omnigent only if the P0 spike measures TTFT p50 ≤ 1.5 s and p95 ≤ 3 s (message-post → first output_text.delta, local server + local runner + the target harness). Until then the routing default is voice: local, text: omnigent-eligible — a spoken "work on X" can still delegate to a substrate thread and Ada narrates progress from Status/Tool chunks, which tolerates seconds of thinking time gracefully.

Preflight (extends M6's VoicePreflight discipline)

  • GET /v1/sessions reachable (server up, auth valid).
  • Session snapshot shows runner_online/host_online for the bound runner — else the turn is refused up front with a actionable message ("runner offline — run omni host …"), never a mid-turn 503.
  • Pinned-version check: server version echoed against the pinned 0.4.x range; a major drift downgrades to Local with a status chunk.
  • ada doctor gains an omnigent section reporting all of the above.

9. Plan of action — phases & acceptance

  • P0Spike (throwaway). Docker-compose Omnigent 0.4.0 + local runner + one harness; drive create→bind→stream→message→deltas→interrupt from a .NET console using a client generated from the pinned openapi.json; capture real SSE frames as fixtures. Exit: e2e turn works; TTFT p50/p95 and token cadence measured and recorded in this doc; captured fixtures checked in. The voice gate (§7) is decided by these numbers.
  • P1OmnigentEngine — text path. Engine + wire client + event fold (fixtures-tested); ada chat --substrate omnigent; route badge shows omnigent · <harness> + cost. Exit: a full CLI conversation on Omnigent with per-turn error isolation; Substrate=Local byte-identical composition proven by tests.
  • P2Threads, reconnect, interrupt. Thread↔session map in SQLite; snapshot+dedupe reconnect; cancellation → wire interrupt; agent-bundle upload + hash refresh. Exit: kill the SSE mid-turn and the turn completes from snapshot; Ctrl-C lands as user_interrupt; a persona edit re-uploads the bundle.
  • P3Elicitation ↔ approval bridge. Elicitations render in Ada's approval UI; accept/decline/cancel resolve; decline cancels the turn; policies (spend cap, intent gate) in the bundle. Exit: a destructive runner command prompts in Ada's window and a decline ends the turn without faulting the thread.
  • P4Voice enablement (behind the gate). If §7 numbers pass: voice routing opt-in; barge-in→interrupt verified against a live server; spoken delegation ("work on X and tell me when done") narrates Status/Tool chunks regardless of the gate. Exit: barge-in stops runner-side generation ≤ 500 ms after VAD start; a delegated task narrates progress and completion.
  • P5Settings & polish. Settings UI section, remote header-auth posture, ada doctor checks, fork-as-retry. Exit: a user configures a remote Omnigent without editing JSON; doctor diagnoses server/runner/auth states distinctly.

11. Out of scope

  • Audio input to Omnigent (upstream "Not Yet"); STT/TTS remain Ada-local permanently under this spec.
  • Exporting Ada's gated local tools as Omnigent client-side tools (recorded as the natural M9.1).
  • Multi-device session sync UI, team governance, cross-conversation search.
  • Any Omnigent-side change — Ada consumes the public API as-is.
  • Cookie-auth (OIDC/accounts) postures.
